# Weekly Digest — 技术周刊深度解读与图文生成

将一期技术周刊拆解为逐篇文章的深度解读，并为每篇文章生成一张信息图。

## 核心流程

```
周刊 URL → 抓取并分类目录 → 逐篇深度阅读 → 提炼核心观点 → 生成信息图 → 输出汇总
```

### Step 1: 抓取周刊目录并分类

使用 `web_fetch` 获取周刊页面内容，解析出所有文章条目。每个条目需要提取：

- **标题**（原文标题）
- **作者**
- **原文链接**
- **周刊编辑给出的简介/推荐语**（如果有的话）
- **所属分类**（如 NEWS、AUTOMATION、TOOLS 等，如果周刊有分类的话）
- **条目类型**（见下方分类规则）

#### 条目分类规则

周刊中的条目并非都是深度文章，需要区分处理：

| 条目类型 | 识别特征 | 处理方式 |
|----------|---------|---------|
| **深度文章** | 有独立链接、有编辑推荐语、内容是观点/教程/实践分享 | 逐篇深度阅读 + 独立信息图 |
| **快讯/简报** | "IN BRIEF" 板块、一两句话的短条目 | 合并为一张"本期快讯速览"信息图 |
| **版本发布** | "RELEASES" 板块、"v1.2.3" 格式的版本号 | 逐个访问发布页/changelog，提取版本亮点，每个生成独立信息图 |
| **赞助商广告** | 标记为 "sponsor"、"sponsored" | 直接跳过，不做解读 |
| **外部推荐** | "Elsewhere"、"Classifieds" 板块 | 合并为一张"延伸阅读推荐"信息图 |

将解析结果整理为一份结构化的文章清单，按上述分类标注每个条目的类型，向用户确认文章数量和列表，然后开始逐篇处理。

### Step 2: 逐篇深度阅读

对分类为**深度文章**的每一篇：

1. 使用 `web_fetch` 访问原文链接，获取文章全文
2. 如果原文过长，分段读取，确保覆盖完整内容
3. **链接失败容错**：如果 `web_fetch` 访问失败（SSL 错误、DNS 解析失败、超时、404、付费墙等），按以下顺序尝试恢复：
   - 首先，使用 `web_search` 搜索文章标题 + 作者名，尝试找到文章的其他可访问镜像（如 daily.dev、Reddit、Hacker News 讨论帖等）
   - 如果找到可访问的镜像或缓存版本，从中获取文章内容
   - 如果仍然无法获取原文，基于周刊中的简介进行解读，并标注"⚠️ 原文未能完整获取，以下解读基于周刊简介"

对于**快讯/简报**、**外部推荐**类条目，无需逐篇访问原文，直接基于周刊页面中的信息进行汇总即可。

对于**版本发布**类条目，逐个使用 `web_fetch` 访问其发布页或 changelog 链接，提取该版本的核心亮点（新特性、破坏性变更、性能提升等），为每个版本发布生成独立的解读文件和信息图。

### Step 3: 提炼核心观点

#### 深度文章的解读格式

为每篇深度文章撰写中文解读，包含以下要素：

- **文章标题**（中文翻译 + 原文标题）
- **作者**
- **一句话总结**：用一句话概括文章的核心主张
- **核心观点**（3-5 个要点）：
  - 文章提出了什么问题或观察
  - 作者的核心论点和论据
  - 关键数据、案例或实践经验
  - 对读者的启发或行动建议
- **关键引用**：文章中最有力的一句原文引用（如果有的话）
- **适合谁读**：简要说明这篇文章最适合什么角色/场景的读者

每篇文章的解读内容保存为独立的 Markdown 文件。文件命名格式：`{序号}-{简短英文标识}.md`，例如 `01-rethinking-testing.md`。

#### 版本发布类条目的解读格式

为每个版本发布撰写独立的中文解读，包含：

- **工具/库名称 + 版本号**（如 "Bun v1.2.15"）
- **一句话总结**：这个版本最值得关注的变化
- **版本亮点**（2-4 个要点）：
  - 新增了什么核心特性
  - 有哪些破坏性变更（breaking changes）
  - 性能或体验上的改进
  - 对开发者的实际影响
- **升级建议**：是否建议立即升级，有无需要注意的迁移事项

文件命名格式：`{序号}-release-{工具名}.md`，例如 `12-release-bun.md`。

#### 合并类条目的解读格式

快讯和外部推荐等合并类条目，各自汇总为一个 Markdown 文件：

- `briefs-roundup.md` — 快讯速览
- `elsewhere-roundup.md` — 延伸阅读推荐

每个条目用简短的中文描述概括要点即可。

### Step 4: 为每篇文章生成信息图

对每篇文章的解读内容，调用 `baoyu-infographic` 生成一张信息图。

#### 布局与风格选择策略

根据文章内容的类型，智能选择最合适的 layout × style 组合：

| 文章类型 | 推荐 Layout | 推荐 Style |
|----------|-------------|------------|
| 观点/思考类 | `hub-spoke` | `craft-handmade` |
| 流程/方法论 | `linear-progression` | `ikea-manual` |
| 对比/辩论类 | `binary-comparison` | `corporate-memphis` |
| 工具介绍/评测 | `structural-breakdown` | `technical-schematic` |
| 趋势/预测 | `winding-roadmap` | `cyberpunk-neon` |
| 实践经验/案例 | `bento-grid` | `bold-graphic` |
| 书评/推荐 | `hierarchical-layers` | `aged-academia` |
| 数据/指标 | `dashboard` | `pop-laboratory` |
| 版本发布/更新 | `structural-breakdown` | `pixel-art` |
| 快讯速览（合并） | `dense-modules` | `morandi-journal` |
| 延伸阅读（合并） | `bento-grid` | `hand-drawn-edu` |

不必死板遵循上表，根据具体文章内容灵活调整。同一期周刊中尽量让不同文章使用不同的 layout × style 组合，保持视觉多样性。

#### 生成命令

```bash
/baoyu-infographic {解读文件路径} --layout {选定layout} --style {选定style} --lang zh
```

### Step 5: 生成总览信息图

所有文章处理完成后，为整期周刊生成一张**总览信息图**，将本期所有文章浓缩在一张图中，让读者一眼看清本期全貌。

总览信息图的内容应包含：
- **周刊名称 + 期数 + 日期**
- **本期主题/编辑寄语**（如果有的话）
- **每篇文章的标题 + 一句话总结**（按分类分组展示）
- **本期亮点标注**（2-3 篇最值得深读的文章用特殊标记突出）
- **趋势关键词**（从本期所有文章中提炼的 3-5 个行业趋势关键词）

总览信息图推荐使用 `dense-modules` 布局 + `retro-pop-grid` 风格，纵向排列（portrait），以容纳更多内容。

```bash
/baoyu-infographic {总览文件路径} --layout dense-modules --style retro-pop-grid --aspect portrait --lang zh
```

将总览解读内容保存为 `overview.md`，信息图保存为 `overview.png`。

### Step 6: 输出汇总

生成一份汇总报告 `README.md`，包含：

1. **周刊概览**：周刊名称、期数、日期、编辑寄语（如果有的话）
2. **总览信息图**：嵌入总览信息图的引用
3. **文章索引**：每篇文章的标题、一句话总结、对应信息图的文件路径
4. **本期亮点**：编辑推荐的 2-3 篇最值得深读的文章及理由
5. **趋势洞察**：从本期所有文章中提炼出的行业趋势或共性话题

## 注意事项

- **跳过广告**：赞助商/广告内容（标记为 sponsor、sponsored 的条目）一律跳过，不做任何解读
- **尊重原文**：解读应忠实于原文观点，不要过度演绎或添加原文没有的论点
- **中文输出**：所有解读内容和信息图文字均使用中文，但保留原文标题作为参考
- **访问限制**：部分文章可能有付费墙或访问限制，遇到这种情况基于周刊简介进行解读即可，不要跳过
- **批量处理**：一期周刊通常有 10-20 篇文章，按顺序逐篇处理，每完成一篇就告知用户进度
- **视觉多样性**：同一期周刊的信息图应使用不同的 layout × style 组合，避免视觉疲劳
- **视频类条目**：如果条目链接指向 YouTube 等视频平台，基于周刊简介进行解读即可，无需尝试获取视频内容
